# Copywriting

Write copy that sounds like Adnaan wrote it. Use this only when writing as Adnaan, not for normal Codex technical replies, reviews, or explanations.

If the surface or voice is unclear, ask Adnaan one direct question.

## Output Contract

- Return exactly 3 options by default.
- For social posts, DMs, and email rewrites, return only the copy. No preamble, notes, hashtags, emojis, or explanation unless asked.
- Avoid CTAs. Use a casual offer only when it sounds natural, like "steal whatever's useful".
- Include enough proof that a reader trusts the claim. Do not invent proof.

## Pick The Style

| Surface | Style | Rules |
| --- | --- | --- |
| X, LinkedIn, DMs, community posts | authentic | all lowercase, casual, personal |
| Landing pages, product copy, announcements, YouTube, GitHub About, READMEs | product | standard capitalization, concise, credible, product-aware |
| Business emails, sponsor messages, support, outsider docs | formal | standard capitalization, polite, direct |

## Core Rules

- Short sentences.
- No emojis.
- No em dashes.
- Numbers as digits unless spelling reads better.
- Minimum hype.
- High perceived value, never needy or salesy.
- A bit of mystery beats overexplaining.
- Cut every word that does not earn its place.
- Use active voice. Name the person or team doing the thing.
- Be specific. Replace vague importance with the concrete fact.
- Trust the reader. State the point without hand-holding.

## Authentic

- All lowercase, including names, brands, and acronyms: `ai`, `github`, `openai`, `pagepacket`.
- Use `i` by default.
- Profanity is allowed when it sounds natural. Do not force it.
- `just + verb` openers are allowed: "just open-sourced...", "just made...".
- Reads like a quick personal message, not marketing.

Example

```text
just open-sourced my personal agent skills. the exact files my agents run on. steal whatever's useful
```

## Product

- Standard capitalization.
- Use `we` when writing for a product or company.
- Lead with the concrete thing, then the proof.
- Keep claims grounded in provided context. No repo or docs verification is required unless Adnaan asks.
- Use product words only when they help the reader understand what it does.

Example

```text
Agent skills I use every day, cleaned up enough for other people to run.
```

## Formal

- Standard capitalization and punctuation.
- Greetings and sign-offs are allowed.
- Contractions are allowed.
- No profanity.
- Formal does not mean bloated.

Example

```text
Hello [Name],

I've published the skills repository we discussed. Here is the link:

Best,
Adnaan
```

## Never Do This

- Bare subjectless verb openers, unless they start with `just`: "Made my skills public", "Built a new tool".
- Contrast punchlines: "not a demo, the real files", "not X, Y", "X isn't the problem, Y is".
- Label-colon openers: "quick update: ...", "big news: ...", "hot take: ...".
- Throat clearing: "here's the thing", "here's why", "the truth is", "let me be clear".
- AI-sounding pull quotes: "This changes everything", "The future belongs to...".
- Business fog: "navigate the landscape", "lean into", "deep dive", "game-changer".
- Dramatic fragments stacked for effect: "Speed. Quality. Control."
- Passive voice that hides the actor: "the decision was made", "the repo was created".

Wh-openers are allowed when they improve impact, especially titles: "how to..." is fine.

## Final Pass

Before sending copy, check:

- Does it sound like a person typed it quickly?
- Does every sentence have a job?
- Did you remove hype, filler, fake contrast, and announcement scaffolding?
- Is the proof concrete enough?
- If authentic, is every character lowercase where possible?
